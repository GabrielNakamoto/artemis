❯ what does GC stand for                                                        

● Garbage Collection — the JVM's automatic memory management process that
  reclaims unused objects. In Java-based FRC robot code (WPILib), the JVM
  periodically pauses execution to free heap memory, which shows up as GCTimeMS
  spikes in the log.

  ---
  Key Findings

  1. Consistent ~5x velocity overshoot — Feed runs at ~95 RPS vs 20 RPS
  setpoint; index at ~57 RPS vs 10 RPS. The ratio is too consistent to be a PID
  tuning issue. This is almost certainly a gear ratio or unit conversion 
  mismatch — the setpoint is in mechanism-level RPS, but the velocity feedback
  is reporting motor-shaft RPS (through an unconverted gear ratio). The control
  is effectively open-loop feedforward.

  2. Voltage clamps at ~2.2V — Consistent across all cycles, not responding to
  velocity error. This confirms the controller is running feedforward-only (no
  closed-loop correction), or the feedback path has the wrong units and the PID
  gain is effectively zero.

  3. Very high index motor currents — Peak 115.9A (FEED 11), regularly 70–88A.
  These are close to stall current for a Falcon 500 and indicate the note is
  loading the index wheel heavily. This likely causes the 2 REVERSE events
  (jams).

  Bottom line: The spindexer PID has a gear ratio not applied to velocity
  feedback. The motors still shoot notes (voltage is consistent and notes do get
   fed), but there's no velocity regulation — current spikes and jams will
  persist until the velocity sensor conversion matches the setpoint units.
